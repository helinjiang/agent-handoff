# 项目愿景（Vision）

AgentRelay 旨在简化真实 AI Coding 场景中的多 Agent 协作问题。

## 问题背景

在 AI Coding 实践中，我们观察到以下痛点：

1. **上下文膨胀** - 单一 Agent 随着任务推进，上下文越来越长，导致质量下降、成本上升
2. **能力差异** - 不同模型/Agent 擅长不同任务（需求澄清、架构设计、代码实现、测试验证），需要分工协作
3. **协作断层** - 多任务（Task）模式下上下文隔离，跨任务协作困难，关键信息容易丢失
4. **交接混乱** - 手动交接不结构化，依赖对话记忆，难以追溯和回放

## 核心愿景

我们希望实现：

- **结构化交接（Handoff）** - 通过产物驱动，上一步输出自动成为下一步输入
- **控制式上下文共享** - 只共享完成任务必要的信息，避免上下文污染
- **可回放的工作流** - 所有决策和变更都有记录，可审计、可复盘
- **可插拔的执行方式** - TRAE-first，但可扩展到其他执行端

## 设计原则

1. **线性优先** - v0.x 只做线性步骤链，保持简单可控，避免过度抽象
2. **产物驱动** - 工作流真相源来自产物（文件/记录），而非对话历史
3. **最小共享** - 只传递必要信息：目标、约束、验收标准、变更摘要、风险点
4. **渐进增强** - 从辅助模式（生成 prompt）开始，逐步演进到半自动、全自动

## 我们不是什么

- **不是 DAG 调度器** - 不做复杂分支/并行调度，保持线性模型
- **不是 UI Agent 框架** - 不追求单智能体自主循环（区别于 OpenClaw 类项目）
- **不是企业平台** - 不做多租户/权限/计费等平台化能力

## 目标用户

- 使用 TRAE 或类似 AI Coding 工具的开发者
- 需要多 Agent 协作的复杂项目
- 希望工作流可追溯、可复盘的团队

## 成功指标

- 一个完整工作流结束后，通过产物即可复现所有关键决策
- 新成员可以通过 events.jsonl 快速理解项目演进历史
- 多 Agent 协作不再依赖"在同一个对话里"

---

AgentRelay 强调**协作效率**，而非**完全自治**。
